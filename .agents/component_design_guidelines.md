## 1. Philosophy & Design Principles

### 1.1 What Spectrum UI Is

Spectrum UI is a **copy-paste component library** built for developers who care about quality. Every component must feel like it was hand-crafted by a senior design engineer at a top-tier product company — not generated by a template engine, not assembled from generic blocks.

The benchmark is not "does it work." The benchmark is: **would a developer at Linear, Vercel, or Resend be proud to ship this?**

### 1.2 Core Principles

**Principle 1: Restraint over decoration.**  
Every visual element must justify its existence. If removing it makes the component cleaner, remove it. Empty space is a design decision, not an oversight.

**Principle 2: Motion as feedback, not performance.**  
Animations exist to communicate state changes to the user. They are never decorative loops, never continuous, never distracting. A user should feel the interface respond — not watch it perform.

**Principle 3: Monochrome first, contrast always.**  
Spectrum UI is a monochrome design system. Color is not the mechanism for emphasis. Contrast, weight, size, and spacing create hierarchy. This is non-negotiable.

**Principle 4: Developer experience is the product.**  
Copy-paste must work on the first try. Props must be typed. Imports must be clean. If a developer has to debug your component before using it, it has failed.

**Principle 5: Accessible by default.**  
Accessibility is not an afterthought. Keyboard navigation, focus management, screen reader support, and ARIA semantics are built in from the start, not added later.

### 1.3 Quality Bar

Before any component or section is considered complete, it must pass this test:

- Does it look better than every free alternative available today?
- Does it handle all edge cases (long text, empty state, loading state, error state)?
- Can a developer copy it into any Next.js + Tailwind project and have it work in under 2 minutes?
- Does it behave correctly on mobile, tablet, and desktop without modification?
- Would you personally use it in a production project?

If the answer to any of these is "no" or "maybe," it is not ready.

## 2. Visual Design System

### 2.1 Aesthetic Direction

Spectrum UI follows a **neutral precision** aesthetic. This is characterized by:

- Ultra-thin borders with low opacity
- Subtle surface layering through background opacity differences, not color
- Glassmorphism used sparingly for overlays and floating elements only
- Sharp, geometric shapes with consistent border radius at each scale
- No gradients on structural elements (buttons, cards, nav) — gradients only as decorative accents in hero sections, and even then, strictly gray/neutral

### 2.2 Surface Hierarchy

There are exactly four surface levels in Spectrum UI. Never add more. Never skip levels.

| Level | Light Mode | Dark Mode | Usage |
|-------|-----------|-----------|-------|
| Base | `bg-white` | `dark:bg-[#0C0C0C]` | Page background |
| Raised | `bg-neutral-50` | `dark:bg-neutral-900` | Cards, panels, sidebars |
| Elevated | `bg-neutral-100` | `dark:bg-neutral-800` | Dropdowns, tooltips, popovers |
| Floating | `bg-white/40 backdrop-blur-md` | `dark:bg-black/35 backdrop-blur-md` | Modals, command palettes, sticky navbars |

**Rule**: A component at one surface level must only contain elements from the same or higher level. Never place a `bg-neutral-50` card inside a `bg-neutral-50` panel — step up to `bg-neutral-100` for inner elements.

### 2.3 Border Standards

All borders must be:
- `1px` thick — never `2px` on structural elements
- Low contrast against their background: `border-neutral-200 dark:border-neutral-800`
- For glassmorphism surfaces: `border-white/20 dark:border-white/10`
- Never colored. No `border-blue-500`, no `border-violet-400`.

Focus rings are the only exception — they use `ring-1 ring-neutral-950 dark:ring-neutral-300` and appear only on `focus-visible`, never `focus`.

## 3. Typography Standards

### 3.1 Font Scale

Spectrum UI uses a strict typographic scale. Do not introduce intermediate sizes.

| Token | Size | Weight | Line Height | Usage |
|-------|------|--------|-------------|-------|
| `display` | `text-5xl` / `text-6xl` | `font-semibold` | `leading-tight` | Hero headlines |
| `heading-1` | `text-4xl` | `font-semibold` | `leading-tight` | Section headers |
| `heading-2` | `text-2xl` | `font-medium` | `leading-snug` | Card titles, sub-sections |
| `heading-3` | `text-xl` | `font-medium` | `leading-snug` | Feature titles |
| `body` | `text-base` | `font-normal` | `leading-relaxed` | Primary body text |
| `body-sm` | `text-sm` | `font-normal` | `leading-relaxed` | Secondary descriptions |
| `caption` | `text-xs` | `font-medium` | `leading-normal` | Labels, timestamps, badges |
| `code` | `text-sm font-mono` | `font-normal` | `leading-relaxed` | Inline code, code blocks |

### 3.2 Text Color Hierarchy

- **Primary text**: `text-neutral-900 dark:text-neutral-100` — headlines, important labels
- **Secondary text**: `text-neutral-600 dark:text-neutral-400` — body copy, descriptions
- **Tertiary text**: `text-neutral-400 dark:text-neutral-600` — hints, placeholders, timestamps
- **Disabled text**: `text-neutral-300 dark:text-neutral-700` — explicitly disabled state only

Never use `text-black` or `text-white` directly. Always use the neutral scale.

### 3.3 Typography in Section Blocks

Hero section text must follow this exact hierarchy:
1. Eyebrow label — `text-xs font-medium uppercase tracking-widest text-neutral-400`
2. Headline — `text-5xl font-semibold tracking-tight text-neutral-900 dark:text-neutral-100`
3. Subheadline — `text-lg text-neutral-500 dark:text-neutral-400 max-w-prose`
4. CTA — Button component (see Button standards below)

This ordering is not a suggestion. Deviation requires explicit design review.

## 4. Spacing & Layout Architecture

### 4.1 Spacing Scale

Use Tailwind's default spacing scale. Avoid arbitrary values (`p-[13px]`) unless absolutely required for pixel-perfect alignment with external constraints. If you find yourself using arbitrary spacing frequently, the component's base spacing assumptions are wrong — fix the structure, not the spacing.

Preferred spacing units for component internals:

- **Micro gaps** (icon-to-label, badge internal): `gap-1.5`, `gap-2`
- **Component internal padding**: `p-3`, `p-4`, `px-4 py-3`
- **Card padding**: `p-5`, `p-6`
- **Section padding**: `py-16 px-6`, `py-24 px-8` for larger viewports

### 4.2 Border Radius Scale

Border radius is assigned by element scale. This is strict.

| Element | Class | Pixels |
|---------|-------|--------|
| Badges, tags, pills | `rounded-full` | Full |
| Inputs, buttons, small widgets | `rounded-xl` | 12px |
| Cards, dropdowns, tooltips | `rounded-2xl` | 16px |
| Modals, large panels, section containers | `rounded-3xl` | 24px |

Never use `rounded-md` or `rounded-lg` — these are reserved for legacy browser defaults and break the visual consistency of the system.

### 4.3 Grid & Container Standards

- Max content width: `max-w-7xl mx-auto`
- Section containers always use: `w-full px-4 md:px-6 lg:px-8`
- Feature grids: `grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-6`
- Two-column layouts: `grid grid-cols-1 lg:grid-cols-2 gap-12 items-center`
- Never hardcode pixel widths on layout containers. Use max-width + auto margins.

## 5. Color System & Tokens

### 5.1 The Monochrome Rule

Spectrum UI is built on a monochrome foundation. This is the single most important visual decision in the system. The reasons:

1. Monochrome scales universally across any project — developers don't need to retheme it
2. It forces hierarchy to be expressed through contrast, weight, and space — stronger design fundamentals
3. It looks premium across light and dark mode without additional work

**The rule**: Structural elements (buttons, cards, inputs, navbars, borders, icons) use only black, white, and the `neutral-100` through `neutral-900` scale.

### 5.2 Allowed Color Usage

Color is permitted only in these specific, limited cases:

| Use Case | Allowed Colors | Example |
|----------|---------------|---------|
| Status indicators | `text-green-600`, `text-red-500`, `text-yellow-500` | Success/error/warning badges |
| External brand logos | Passthrough — don't override | Tech stack logos in a grid |
| User-generated content | Passthrough — don't override | Avatar backgrounds |
| Decorative blobs in hero sections | Single-hue, low-opacity, blurred: `bg-neutral-300/20 blur-3xl` | Background atmosphere |

Everything else is neutral. If you feel the urge to add a colored border or accent, ask yourself: can contrast or weight solve this instead?

### 5.3 Dark Mode Requirements

Every component must support dark mode. Rules:

- Use the `dark:` prefix on every color utility — never a single color without its dark counterpart
- Test dark mode explicitly before marking a component complete
- Dark mode backgrounds are NOT simply inverted light mode — they are slightly elevated from pure black: `#0C0C0C` base, `neutral-900` surfaces
- Text in dark mode should never be pure white (`text-white`) — use `dark:text-neutral-100` to maintain the premium muted quality

## 6. Animation & Motion Standards

### 6.1 The Motion Contract

Animations in Spectrum UI follow a strict contract:

1. **Triggered, not looped** — animations fire in response to user actions or component mount. They never loop continuously.
2. **Spring, not duration** — use `type: "spring"` with physics presets. Duration-based easing (linear, ease-in-out) is only acceptable for opacity-only transitions where spring physics have no visual benefit.
3. **Fast and settled** — users should not wait for animations. They should feel the interface respond. Spring animations must settle within 400ms in all standard interactions.
4. **Purposeful** — every animation communicates something: a state change, a response to input, an element entering or leaving. Decorative animations that exist only to look impressive are banned.

### 6.2 Spring Physics Presets

These are the canonical spring presets. Do not invent new ones without design review.

```typescript
// Preset 1: Tactile — for buttons, toggles, hover scaling
const SPRING_TACTILE = {
  type: "spring",
  stiffness: 400,
  damping: 25,
  mass: 1,
}

// Preset 2: Fluid — for tabs, sliders, drawer handles, expandable panels
const SPRING_FLUID = {
  type: "spring",
  stiffness: 300,
  damping: 30,
}

// Preset 3: Entrance — for modals, overlays, command palettes
const SPRING_ENTRANCE = {
  type: "spring",
  stiffness: 260,
  damping: 20,
}

// Preset 4: Snappy — for tooltips, badges, micro-pop effects
const SPRING_SNAPPY = {
  type: "spring",
  stiffness: 500,
  damping: 28,
}
```

### 6.3 Standard Animation Patterns

**Button hover/tap:**
```typescript
<motion.button
  whileHover={{ scale: 1.02 }}
  whileTap={{ scale: 0.98 }}
  transition={SPRING_TACTILE}
>
```

**Modal entrance:**
```typescript
<motion.div
  initial={{ opacity: 0, y: 12, scale: 0.98 }}
  animate={{ opacity: 1, y: 0, scale: 1 }}
  exit={{ opacity: 0, y: 8, scale: 0.98 }}
  transition={SPRING_ENTRANCE}
>
```

**Accordion / expandable panel:**
```typescript
<motion.div
  layout
  initial={false}
  animate={{ height: isOpen ? "auto" : 0 }}
  transition={SPRING_FLUID}
  className="overflow-hidden"
>
```

**Tabs active indicator (layoutId pattern):**
```typescript
// Active pill slides between tabs using layoutId
{isActive && (
  <motion.div
    layoutId="tab-indicator"
    className="absolute inset-0 bg-neutral-900 dark:bg-neutral-100 rounded-xl"
    transition={SPRING_FLUID}
  />
)}
```

**List item stagger entrance:**
```typescript
const containerVariants = {
  hidden: {},
  visible: {
    transition: {
      staggerChildren: 0.06,
    },
  },
}

const itemVariants = {
  hidden: { opacity: 0, y: 8 },
  visible: { opacity: 1, y: 0, transition: SPRING_ENTRANCE },
}
```

### 6.4 AnimatePresence Requirements

Any component that conditionally mounts/unmounts animated children must wrap them in `<AnimatePresence>`. This includes:

- Modals and drawers
- Dropdowns and popovers
- Toast notifications
- Command palette results
- Conditional error messages
- Tooltip content

```typescript
import { AnimatePresence, motion } from "framer-motion"

<AnimatePresence>
  {isOpen && (
    <motion.div
      key="modal"
      initial={{ opacity: 0, scale: 0.97 }}
      animate={{ opacity: 1, scale: 1 }}
      exit={{ opacity: 0, scale: 0.97 }}
      transition={SPRING_ENTRANCE}
    />
  )}
</AnimatePresence>
```

### 6.5 Reduced Motion

Always respect the user's OS-level reduced motion preference:

```typescript
import { useReducedMotion } from "framer-motion"

function MyComponent() {
  const shouldReduceMotion = useReducedMotion()
  
  const transition = shouldReduceMotion
    ? { duration: 0 }
    : SPRING_ENTRANCE
}
```

## 7. Component Architecture Rules

### 7.1 File Structure

Each component lives in a single file. The file must contain:

1. All imports at the top
2. All TypeScript interfaces and types immediately after imports
3. Any local constants (spring presets, class maps) after types
4. The main component export
5. Any sub-components used only within this file (not exported)
6. No default exports — all components use named exports

```typescript
// 1. Imports
import { motion, AnimatePresence } from "framer-motion"
import { cn } from "@/lib/utils"
import type { ReactNode } from "react"

// 2. Types
export interface CardProps {
  title: string
  description?: string
  className?: string
  children?: ReactNode
}

// 3. Constants
const SPRING = { type: "spring", stiffness: 300, damping: 30 } as const

// 4. Main component (named export)
export function Card({ title, description, className, children }: CardProps) {
  return (
    <div className={cn("rounded-2xl border border-neutral-200 dark:border-neutral-800 bg-white dark:bg-neutral-900 p-6", className)}>
      <h3 className="text-xl font-medium text-neutral-900 dark:text-neutral-100">{title}</h3>
      {description && (
        <p className="mt-2 text-sm text-neutral-500 dark:text-neutral-400">{description}</p>
      )}
      {children}
    </div>
  )
}
```

### 7.2 Props Design Rules

**Rule 1: No `any` types. Ever.**  
If you don't know the type, figure it out. Use `unknown` and narrow it. Use generics. There is no acceptable reason for `any` in a published component.

**Rule 2: Export all interfaces.**  
Developers often need to extend or type-check against your props. Always export.

**Rule 3: Provide sensible defaults.**  
Optional props must have defaults that produce a usable, visually complete component without any configuration.

**Rule 4: Accept and forward `className`.**  
Every component that renders a DOM element must accept `className` and merge it with `cn()`. This is non-negotiable — it's how developers customize without forking.

**Rule 5: Never hard-code content.**  
No component should have a fixed string that a developer cannot override. All copy is a prop. All icons can be swapped. All colors follow the token system.

```typescript
// Wrong
export function Badge() {
  return <span className="...">New</span>
}

// Correct
export interface BadgeProps {
  label: string
  className?: string
}

export function Badge({ label, className }: BadgeProps) {
  return <span className={cn("...", className)}>{label}</span>
}
```

### 7.3 State Management within Components

- Use `useState` for local UI state (open/closed, active tab index, hover)
- Use `useRef` for DOM references, previous values, and timers — never as a substitute for state
- Use `useCallback` on event handlers passed as props to avoid unnecessary re-renders
- Use `useMemo` on computed values that are genuinely expensive — not on every derived value
- Never use component-level state for data that belongs in the application (forms, server data, global UI)

### 7.4 Event Handler Patterns

```typescript
// Always check existence before calling prop handlers
const handleClick = useCallback((e: React.MouseEvent<HTMLButtonElement>) => {
  e.preventDefault()
  onPress?.(e)
}, [onPress])

// Escape key handler for overlays
useEffect(() => {
  if (!isOpen) return
  
  const handleKeyDown = (e: KeyboardEvent) => {
    if (e.key === "Escape") onClose()
  }
  
  document.addEventListener("keydown", handleKeyDown)
  return () => document.removeEventListener("keydown", handleKeyDown)
}, [isOpen, onClose])
```

### 7.5 Ref Forwarding

Components that wrap native inputs, buttons, or focusable elements must use `forwardRef` so developers can control focus programmatically:

```typescript
export const Input = React.forwardRef<HTMLInputElement, InputProps>(
  ({ className, ...props }, ref) => {
    return (
      <input
        ref={ref}
        className={cn("base-input-classes", className)}
        {...props}
      />
    )
  }
)
Input.displayName = "Input"
```

## 8. Section Block Standards

Section blocks are full-width, page-ready components that combine multiple UI primitives into a cohesive layout. They are not pages — they are composable units. Every section block must be independently usable, independently styled, and follow all component standards above.

### 8.1 Required Section Types

The following section types must be available in Spectrum UI. Each is a distinct section block with multiple variants.

**Navigation**
- Fixed top navbar with blur + border on scroll
- Sidebar nav (collapsible, with group headers)
- Mobile hamburger → sheet drawer navigation
- Command-bar nav (keyboard-driven, `⌘K` trigger)

**Hero Sections**
- Centered hero with eyebrow + headline + CTA pair + background blob
- Split hero (text left, visual/mockup right)
- Terminal-style hero (animated code typing effect)
- Gradient-free, full-screen hero with scrolling marquee below
- Video background hero with dark overlay and centered content

**Feature Sections**
- 3-column icon + text feature grid
- Alternating feature rows (image/visual alternates left-right)
- Bento grid feature layout (asymmetric card grid)
- Comparison table (free vs paid vs enterprise)

**Social Proof**
- Logo marquee (infinite scroll of company logos, grayscale)
- Testimonial grid (masonry-style quote cards)
- Single large testimonial with avatar + company
- Stats row (4 numbers with labels, subtle entry animation)

**Pricing**
- 3-tier pricing cards with monthly/annual toggle
- Feature comparison table below pricing cards
- Single highlighted plan (for simple SaaS)

**Content & Blog**
- Blog post card grid (3-column with image, tag, title, excerpt)
- Newsletter CTA section (email input + value proposition)
- Documentation sidebar + content layout

**Call to Action**
- Full-width dark CTA band (headline + button, no noise)
- Split CTA (text left, form right)
- Floating CTA bar (fixed bottom on scroll)

**Footer**
- Multi-column footer with logo, links, social icons
- Minimal single-row footer
- Footer with newsletter embed

### 8.2 Section Block File Requirements

Each section block must include:

1. A named export for the full section component
2. Named exports for each major sub-component (so developers can remix)
3. A `data` prop or exported `defaultData` constant with all copy — no hardcoded strings
4. Full TypeScript types for the data shape
5. Dark mode support
6. Mobile responsiveness tested at 375px, 768px, 1280px, 1440px
7. A comment at the top of the file listing all required dependencies

```typescript
/**
 * Spectrum UI — PricingSection
 * 
 * Dependencies: framer-motion, @/lib/utils
 * Optional: lucide-react (for check icons)
 * 
 * Usage:
 *   import { PricingSection } from "@/components/spectrum/pricing-section"
 *   <PricingSection plans={myPlans} />
 */
```

### 8.3 Section Spacing Standards

Every section block must use consistent vertical rhythm:

- **Section top/bottom padding**: `py-16 md:py-24` minimum. `py-24 md:py-32` for hero and major sections.
- **Section header bottom margin**: `mb-12 md:mb-16` before the content grid
- **Card gaps**: `gap-6` for 3-column grids, `gap-8` for 2-column, `gap-4` for dense lists
- **Section max-width wrapper**: `max-w-7xl mx-auto w-full px-4 md:px-6 lg:px-8`

## 9. Accessibility Requirements

### 9.1 Semantic HTML

The correct HTML element is always preferred over a styled `div`. This is not optional.

| Element | Use For |
|---------|---------|
| `<button>` | Any clickable action |
| `<a>` | Navigation to a URL |
| `<nav>` | Navigation sections |
| `<main>` | Primary page content |
| `<section>` | Thematic content groups with headings |
| `<article>` | Self-contained content (blog posts, cards) |
| `<header>`, `<footer>` | Document-level header and footer |
| `<h1>`–`<h6>` | Heading hierarchy, one `h1` per page |

Never use `<div onClick={...}>` when `<button>` is appropriate. Never use `<span>` as a link. These are blocking PR issues.

### 9.2 ARIA Requirements

- All icon-only buttons must have `aria-label`
- All form inputs must have an associated `<label>` (either visible or `sr-only`)
- Modals must use `role="dialog"` and `aria-modal="true"` with a visible title via `aria-labelledby`
- Expanding components (accordion, dropdown) must use `aria-expanded` on the trigger
- Loading states must use `aria-live="polite"` on the status container
- Decorative images must have `alt=""`; informational images must have descriptive alt text

### 9.3 Keyboard Navigation

The following keyboard interactions are mandatory for relevant components:

| Component | Required Keys |
|-----------|--------------|
| Modal / Drawer | `Escape` to close, focus trap inside |
| Dropdown / Select | `Arrow Up/Down` to navigate, `Enter` to select, `Escape` to close |
| Tabs | `Arrow Left/Right` to switch tabs, `Home`/`End` for first/last |
| Command Palette | `Arrow Up/Down` to navigate results, `Enter` to execute, `Escape` to close |
| Accordion | `Enter`/`Space` on trigger to toggle |
| Toast | `Escape` or close button to dismiss |

### 9.4 Focus Management

- Modals and drawers must move focus to the first interactive element inside when opened
- When a modal closes, focus must return to the trigger that opened it
- Focus rings must be visible at all times when using keyboard navigation: `focus-visible:ring-1 focus-visible:ring-neutral-950 dark:focus-visible:ring-neutral-300 focus-visible:outline-none`
- Never use `outline: none` without providing an alternative visible focus indicator

### 9.5 Touch & Mobile

- Minimum tap target size: `44px × 44px` — use padding to meet this, not fixed height/width
- All hover interactions must have a tap equivalent on mobile
- Modals must become bottom drawers on mobile viewports (`< 640px`) using `vaul` or equivalent
- Horizontal scrollable content must show scroll indicators or have `overflow-x-auto` with `-webkit-overflow-scrolling: touch`

## 10. AI-Assisted Development Protocol

When using AI tools (Claude, Cursor, GitHub Copilot, or similar) to generate or assist with Spectrum UI components, the following protocol is mandatory.

### 10.1 The AI System Prompt

Always begin an AI session with this exact system prompt when generating Spectrum UI components:

```
You are a senior design engineer contributing to Spectrum UI — a monochrome, 
high-fidelity React component library built with Next.js, Tailwind CSS, and 
Framer Motion.

Rules you must strictly follow:
1. No colors except neutral-100 through neutral-900, black, and white.
   No purple, violet, indigo, blue, or any AI-default accent colors.
2. All animations use Framer Motion spring physics. No CSS transitions for
   layout or transform animations.
3. Border radius: rounded-xl for buttons/inputs, rounded-2xl for cards,
   rounded-3xl for modals. Nothing else.
4. Every component accepts a className prop merged with cn() from @/lib/utils.
5. Full TypeScript — no any types, all interfaces exported.
6. Dark mode on every color utility using dark: prefix.
7. No emojis anywhere in the component.
8. No hardcoded strings — all copy is a prop.
9. Semantic HTML — button for actions, a for links, correct heading hierarchy.
10. Every interactive component supports keyboard navigation.
```

### 10.2 AI Output Review Checklist

After receiving AI-generated code, run through this checklist before committing:

- [ ] No violet/purple/indigo/blue accent colors anywhere in the code
- [ ] No emoji characters in any string, label, or JSX
- [ ] No `any` TypeScript types
- [ ] All interfaces are exported
- [ ] `className` prop present and merged with `cn()`
- [ ] Every color utility has a `dark:` counterpart
- [ ] Animations use `type: "spring"` — not `duration` for transforms
- [ ] `AnimatePresence` wraps any conditionally rendered animated children
- [ ] Semantic HTML elements used (not `div` for interactive elements)
- [ ] No hardcoded copy — all text is from props

### 10.3 Prompting for Section Blocks

When prompting AI for full section blocks, provide the data shape first:

```
Build a Spectrum UI PricingSection component that accepts this data:

interface PricingPlan {
  name: string
  price: { monthly: number; annual: number }
  description: string
  features: string[]
  highlighted: boolean
  ctaLabel: string
}

interface PricingSectionProps {
  eyebrow?: string
  headline: string
  subheadline?: string
  plans: PricingPlan[]
  className?: string
}

Follow all Spectrum UI guidelines: monochrome only, Framer Motion spring 
animations, dark mode, full TypeScript, no emojis, semantic HTML, 
rounded-2xl cards.
```

This approach reduces hallucinated color choices and generic patterns by 80%.

### 10.4 AI Red Flags — Auto-Reject

If AI generates any of the following, reject the output and re-prompt:

- `className="... bg-violet-500 ..."` or any non-neutral color on structural elements
- `transition={{ duration: 0.3, ease: "easeInOut" }}` on scale/position animations
- Generic copy: "Elevate your workflow", "Seamless experience", "Unlock the power of"
- `✨`, `🚀`, `⚡`, `🔥` or any emoji in JSX
- `type: any` anywhere
- Missing `dark:` counterpart on any background or text color
- `onClick={() => {}}` on a `<div>` where a `<button>` is appropriate
- Looping animations (`repeat: Infinity` on anything other than loading spinners)

## 11. Code Quality Checklist

This checklist must be completed before any PR is opened or component is marked as ready.

### Visual Quality

## 12. Anti-Patterns — Strictly Forbidden

The following are categorically banned in Spectrum UI. A PR containing any of these will be closed without review until corrected.

### 12.1 Visual Anti-Patterns

**Forbidden colors on structural elements:**
- `bg-violet-*`, `bg-purple-*`, `bg-indigo-*`, `bg-blue-*` on any structural element
- `border-violet-*`, `ring-violet-*` on interactive focus or hover states
- Gradient backgrounds on buttons, cards, or navigation (`bg-gradient-to-r`)
- Box shadows with colored glow: `shadow-[0_0_20px_rgba(139,92,246,0.3)]`

**Forbidden layout choices:**
- `rounded-md` or `rounded-lg` — use `rounded-xl` minimum
- `text-black` or `text-white` directly — use `neutral-900` / `neutral-100`
- Pixel-level arbitrary Tailwind spacing without justification: `p-[17px]`

**Forbidden decorative choices:**
- Any emoji in component code: `✨`, `🚀`, `⚡`, `🎨`, `🔥`, `💡`, `🌟`
- Continuous looping animations (particles, floating elements, breathing glows)
- Gradient noise or SVG noise textures on card surfaces
- Drop shadows on interactive cards on hover — use border opacity change instead

### 12.2 Copy Anti-Patterns

Never use these phrases or their derivatives:

- "Elevate your workflow"
- "Seamless experience"  
- "Next-generation"
- "Unlock the power of"
- "Beautiful by default"
- "Supercharge your"
- "Best-in-class"
- "Industry-leading"

Use literal, specific, developer-oriented copy instead:
- "Keyboard-driven command palette with fuzzy search"
- "Responsive data table with sorting and pagination"
- "Copy-paste auth form with validation"

### 12.3 Code Anti-Patterns

```typescript
// ❌ Never use any
const handleData = (data: any) => {}

// ❌ Never hardcode strings
<Button>Get Started</Button>

// ❌ Never use div for interactive elements
<div onClick={handleClick}>Click me</div>

// ❌ Never use CSS transition for spring animations
transition={{ duration: 0.3, ease: "easeInOut" }}

// ❌ Never ignore dark mode
<div className="bg-white text-neutral-900">

// ❌ Never use emoji
<span>✨ New feature</span>

// ❌ Never loop animations decoratively
animate={{ rotate: 360 }}
transition={{ repeat: Infinity, duration: 2 }}
```

## 13. Naming Conventions

### 13.1 Component Names

- PascalCase for all components: `CommandPalette`, `PricingSection`, `TestimonialCard`
- Descriptive and specific — no generic names: `Card` → `FeatureCard`, `TestimonialCard`, `BlogCard`
- Section blocks use the `Section` suffix: `HeroSection`, `PricingSection`, `FeaturesSection`
- Sub-components use the parent as prefix: `PricingCard`, `PricingToggle`, `PricingFeatureRow`

### 13.2 File Names

- kebab-case for all files: `command-palette.tsx`, `pricing-section.tsx`
- One component per file (sub-components used only within the file are allowed in the same file)
- Index files only for folders with multiple exports that belong together

### 13.3 Prop Names

- `onX` for event callbacks: `onClose`, `onSelect`, `onChange`
- `isX` or `hasX` for boolean states: `isOpen`, `isDisabled`, `hasError`
- `defaultX` for uncontrolled defaults: `defaultOpen`, `defaultValue`
- `className` (never `class`, never `containerClassName` unless truly a container-specific override)

### 13.4 CSS Class Conventions

- Never write custom CSS classes for things Tailwind can handle
- Custom CSS (in `.css` files or `<style>`) only for: custom scrollbar styles, CSS custom properties, `@keyframes` that Tailwind cannot express
- `cn()` is the only method for conditional class application — no template literals for class names

## 14. Component Review Process

### 14.1 Self-Review (Required Before PR)

Run through the full checklist in Section 11 yourself. Every item. Do not open a PR with known failures — document them and explicitly mark them as known issues with a follow-up ticket.

### 14.2 Design Review Criteria

A reviewer checks these in order:

1. **Does it look premium?** — Hold it against a Linear or Vercel component mentally. Is the gap obvious?
2. **Is the monochrome rule respected?** — Scan for any non-neutral colors on structural elements
3. **Are animations correct?** — Spring physics, settled within 400ms, no loops
4. **Is dark mode complete?** — Every color has a dark counterpart
5. **Is copy developer-appropriate?** — No generic marketing language
6. **Is keyboard navigation complete?** — Tab through the component manually

### 14.3 Component Graduation Levels

Components progress through these levels:

| Level | Criteria |
|-------|----------|
| **Draft** | Functional, visual design in progress |
| **Beta** | Visual complete, dark mode done, basic a11y |
| **Stable** | All checklist items passed, keyboard nav, mobile tested |
| **Showcase** | Used in a real project, documented with variants, has Figma file |

Only `Stable` and `Showcase` components ship in releases. `Draft` and `Beta` components may be available in the docs under an explicit beta label with a warning.

## Appendix B — The `cn()` Utility

Always use this exact implementation. Do not use alternatives.

```typescript
// @/lib/utils.ts
import { type ClassValue, clsx } from "clsx"
import { twMerge } from "tailwind-merge"

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs))
}
```
